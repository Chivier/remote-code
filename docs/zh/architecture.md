# 架构概览

Remote Code 采用两层架构设计：**Head Node**（本地控制节点）和 **Daemon**（远程守护进程），通过 SSH 隧道安全通信。

## 数据流

一条用户消息从发出到收到回复的完整流程：

```
1. 用户在 Discord/Telegram 发送消息
2. Bot 接收消息，查找 channel 对应的活跃会话 (SessionRouter)
3. 通过 SSHManager 确保 SSH 隧道已建立
4. DaemonClient 通过 SSH 隧道发送 JSON-RPC 请求 (session.send)
5. Daemon 的 Express 服务器接收请求
6. SessionPool 生成 `claude --print <message> --output-format stream-json` 子进程
7. Claude CLI 处理消息，输出 JSON-lines 流
8. SessionPool 解析 JSON-lines，转换为 StreamEvent
9. Server 通过 SSE 将事件流式推送回 Head Node
10. DaemonClient 解析 SSE 事件流
11. BotBase 将事件格式化并实时更新 Discord/Telegram 消息
12. 用户在聊天窗口中看到 Claude 的实时回复
```

## 关键设计决策

### 每消息生成进程（Per-Message Spawn）

系统采用每条消息单独生成一个 `claude --print` 进程的架构，而非维护长期运行的 Claude 进程。

**原因**：Claude CLI 2.1.76 不支持在 `--print` 模式之外使用 `--input-format stream-json`。因此我们为每条消息启动一个新的 `claude --print <msg> --output-format stream-json --verbose` 进程。

**会话连续性**：通过 `--resume <sdkSessionId>` 参数维持对话上下文。每次 Claude 返回结果时会包含 `session_id`，系统将其保存并在下次调用时使用。

**进程生命周期**：
1. 用户发送消息 → 启动 `claude --print` 进程
2. 进程处理完成 → 自动退出（exit code 0）
3. 下一条消息 → 启动新进程（带 `--resume`）

### SSH 隧道安全模型

Daemon 绑定到 `127.0.0.1`（仅本地回环地址），不对外暴露任何端口。所有通信通过 SSH 端口转发进行：

```
Head Node (localhost:19100) ---SSH 隧道---> 远程 (127.0.0.1:9100)
```

这意味着：
- Daemon 无需任何认证机制
- 网络上的其他机器无法直接访问 Daemon
- 所有数据在 SSH 通道中加密传输
- 安全性完全依赖 SSH 密钥/密码认证

### SQLite 会话状态持久化

Head Node 使用 SQLite 数据库存储会话状态，包含两张表：

- **sessions** — 当前活跃和已分离的会话（每个 channel 最多一个活跃会话）
- **session_log** — 历史会话记录（用于会话恢复查询）

这保证了即使 Head Node 重启，也能恢复之前的会话映射关系。

### SSE 流式响应

Daemon 使用 SSE（Server-Sent Events）将 Claude 的响应流式推送给 Head Node：

- `partial` 事件提供实时文本增量，实现打字效果
- 30 秒心跳 `ping` 防止 SSH 隧道和 HTTP 连接超时
- 客户端断连时自动缓存事件，重连后回放
- `[DONE]` 标记结束 SSE 流

### 消息队列机制

每个会话维护一个消息队列（MessageQueue），具有三重职责：

1. **用户消息缓冲** — 当 Claude 正在处理一条消息时，后续消息自动排队
2. **响应缓冲** — 当 SSH 连接中断时，缓存 Daemon 产生的响应事件
3. **连接状态跟踪** — 追踪客户端连接状态，决定是否缓存响应

## 多机器支持

系统支持同时连接多台远程机器：

- 每台机器有独立的 SSH 隧道和本地端口
- SessionRouter 记录每个聊天频道绑定到哪台机器
- SSHManager 维护连接池，自动重连断开的隧道
- 支持通过跳板机（ProxyJump）访问内网机器
